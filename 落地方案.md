# LLM Wiki 落地实践指南

## 快速开始

### 1. 创建项目结构

```bash
mkdir -p llm-wiki/{raw/assets,wiki/{entities,concepts,sources,synthesis}}
cd llm-wiki
git init
```

最终目录结构：

```
llm-wiki/
├── CLAUDE.md              # Schema：LLM 行为规范
├── raw/                   # 原始资料层（只读）
│   ├── article-1.md
│   ├── paper-2.pdf
│   └── assets/            # 本地图片
├── wiki/                  # Wiki 层（LLM 全权管理）
│   ├── index.md           # 内容目录
│   ├── log.md             # 操作日志
│   ├── entities/          # 实体页（人物、组织、产品…）
│   ├── concepts/          # 概念页（主题、理论、方法…）
│   ├── sources/           # 来源摘要页
│   └── synthesis/         # 综合分析页
└── .git/                  # 版本控制
```

### 2. 创建 Schema 文件（CLAUDE.md）

这是整个系统最关键的文件，它让 LLM 成为纪律化的 wiki 管理员而非通用聊天机器人。

```markdown
# LLM Wiki Schema

## 角色
你是我的知识库管理员。你负责 wiki/ 目录下所有文件的创建、更新和维护。
我只负责提供来源和提出问题。

## 目录约定
- `raw/` — 原始来源，只读，不可修改
- `wiki/entities/` — 实体页（人物、组织、项目等）
- `wiki/concepts/` — 概念页（理论、方法、技术等）
- `wiki/sources/` — 来源摘要页
- `wiki/synthesis/` — 综合分析页
- `wiki/index.md` — 内容目录
- `wiki/log.md` — 操作日志

## 页面格式
每个 wiki 页面必须包含：
- YAML frontmatter（tags, date, sources）
- 使用 [[wiki links]] 做内部链接
- 明确的结构（标题、摘要、正文、相关链接）

### 实体页模板
---
tags: [entity]
date: YYYY-MM-DD
sources: []
---
# 实体名称

> 一句话定义

## 概述

## 关键信息

## 相关链接
- [[相关概念]]
- [[相关来源]]

### 概念页模板
---
tags: [concept]
date: YYYY-MM-DD
sources: []
---
# 概念名称

> 一句话概括

## 核心思想

## 细节

## 相关链接

### 来源摘要页模板
---
tags: [source]
date: YYYY-MM-DD
source_file: raw/xxx.md
---
# 来源标题

## 摘要

## 关键要点

## 影响的 wiki 页面
- [[实体/概念页]]

## 摄取流程
1. 阅读 raw/ 中的新来源
2. 与我讨论关键要点，确认强调什么
3. 在 wiki/sources/ 创建摘要页
4. 创建或更新相关实体页、概念页（通常涉及 10-15 个页面）
5. 更新 wiki/index.md
6. 在 wiki/log.md 追加记录，格式：`## [YYYY-MM-DD] ingest | 标题`

## 查询流程
1. 先读 wiki/index.md 找到相关页面
2. 阅读相关页面后综合回答，附 wiki 内部引用
3. 有价值的回答（比较分析、新发现）沉淀为新的 synthesis/ 页面
4. 更新 index.md

## 巡检流程
检查以下内容：
- 页面间的矛盾
- 被新来源取代的过时信息
- 没有入站链接的孤立页面
- 被提及但缺独立页面的概念
- 缺失的交叉引用
- 可通过搜索填补的数据缺口

## index.md 格式
按类别组织，每项包含：链接 + 一行摘要 + 可选元数据
示例：
### 实体
- [[Entity A]] - 一句话描述 (3 sources, updated 2026-04-10)

## log.md 格式
追加式，每条格式：`## [YYYY-MM-DD] 操作类型 | 标题`
操作类型：ingest | query | lint | update
```

### 3. 初始化核心文件

**wiki/index.md：**

```markdown
# Wiki Index

## 实体

## 概念

## 来源

## 综合分析
```

**wiki/log.md：**

```markdown
# Wiki Log

## [2026-04-10] init | Wiki 初始化
```

### 4. 开始使用

用 Claude Code 打开项目目录，开始丢资料：

```bash
cd llm-wiki
claude
```

## 实操工作流

### 日常三件事

#### 摄取（添加新资料时）

```
你：我把这篇文章放到了 raw/ 里，帮我处理一下
LLM：阅读 → 讨论要点 → 创建摘要 → 更新相关页面 → 更新 index/log
```

#### 查询（有问题时）

```
你：X 和 Y 之间是什么关系？
LLM：读 index → 定位页面 → 综合回答 → 沉淀有价值的发现
```

#### 巡检（定期，建议每周一次）

```
你：帮我检查一下 wiki 的健康状况
LLM：扫描全部页面 → 报告矛盾/孤立页/缺失引用 → 建议改进
```

## 进阶技巧

### 图片本地化

1. Obsidian 设置 → 文件与链接 → 附件文件夹路径设为 `raw/assets/`
2. 设置 → 快捷键 → 搜索 "Download" → 绑定快捷键（如 `Ctrl+Shift+D`）
3. Clip 文章后按快捷键，所有图片自动下载到本地

> LLM 无法直接读取 Markdown 中的内联图片，变通方法是让 LLM 先读文本，再单独查看引用的图片。

### 搜索增强（规模增长后）

当 wiki 超过 100 个来源、数百个页面时，index.md 可能不够用。使用 [qmd](https://github.com/tobi/qmd)：

```bash
# 安装
go install github.com/tobi/qmd@latest

# 构建索引
qmd index wiki/

# 搜索
qmd search "检索关键词"
```

qmd 同时提供 CLI 和 MCP Server，LLM 可以直接调用。

### 输出格式多样化

LLM 的回答不必只是 Markdown：

| 格式 | 用途 |
|------|------|
| Markdown 页面 | 默认输出 |
| 对比表格 | 并排比较实体/概念 |
| Marp 幻灯片 | 从 wiki 内容生成演示文稿 |
| matplotlib 图表 | 可视化趋势和关系 |
| HTML Canvas | 交互式内容 |

## 最佳实践

1. **从小处开始** — 选一个主题（你正在研究的领域），不要上来就搞全知百科
2. **Schema 要迭代** — CLAUDE.md 不用一次写完美，用中发现问题就更新
3. **逐条摄入** — 每次处理一个来源，保持参与感，引导 LLM 关注重点
4. **好的回答要沉淀** — 对比分析、新发现不要浪费在聊天历史里
5. **定期巡检** — 每周一次 lint，保持 wiki 健康
6. **用 git 追踪** — wiki 天然是 git 仓库，享受版本历史和协作能力
7. **用 Obsidian 浏览** — 图谱视图是最好的方式看 wiki 的"形状"

## 与其他方案的对比

| 维度 | 传统 RAG | 笔记应用 | LLM Wiki |
|------|----------|----------|----------|
| 知识积累 | 无（每次重建） | 有（手动维护） | 有（自动维护） |
| 交叉引用 | 无 | 需手动建立 | LLM 自动建立 |
| 维护成本 | 低 | 高（人类放弃的主因） | 趋近于零 |
| 知识复利 | 无 | 慢（受限于人的精力） | 快（LLM 不知疲倦） |
| 矛盾检测 | 无 | 靠人发现 | LLM 主动标记 |
| 个性化 | 低 | 高 | 高 |
