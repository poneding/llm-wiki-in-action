# LLM Wiki 核心思想

> 来源：[Andrej Karpathy - LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

## 核心问题：传统 RAG 没有积累

大多数人使用 LLM 处理文档的方式是 **RAG**：上传文件，查询时检索相关片段，拼凑回答。

```
每次提问 → 检索原始文档 → 重新拼凑 → 一次性回答 → 知识消失
```

这意味着：

- 一个需要综合五篇文章的问题，LLM 每次都要重新找、重新组合
- 交叉引用永远不会建立
- 矛盾永远不会被标记
- **知识永远不会沉淀**

NotebookLM、ChatGPT 文件上传、大多数 RAG 系统都是这样工作的。

---

## 核心理念：让 LLM 持续构建和维护一个持久的 Wiki

不是从原始文档检索，而是让 LLM **增量构建和维护一个持久化的 wiki** —— 一组结构化、相互链接的 Markdown 文件。

```
添加新来源 → LLM 阅读 → 提炼 → 更新 wiki 中 10-15 个相关页面 → 知识沉淀
```

> **Wiki 是一个持久的、可复利的知识产物。**

- 交叉引用已经建好了
- 矛盾已经被标记了
- 综合已经反映了你读过的所有内容
- 每添加一个新来源、每问一个问题，wiki 都会变得更丰富

### 一个类比

> Obsidian 是 IDE，LLM 是程序员，Wiki 是代码库。

你从不动笔写 wiki —— LLM 全权负责创建和维护。你负责策划来源、引导分析、提出好问题。LLM 负责所有苦力活：总结、交叉引用、归档、簿记。

## 三层架构

```
┌─────────────────────────────────────────┐
│           Schema (CLAUDE.md)            │  ← 配置：告诉 LLM 怎么做
├─────────────────────────────────────────┤
│              Wiki (wiki/)               │  ← LLM 生成的结构化 Markdown
├─────────────────────────────────────────┤
│          Raw Sources (raw/)             │  ← 原始资料，只读不可变
└─────────────────────────────────────────┘
```

| 层 | 说明 | 谁负责 |
|---|---|---|
| **Raw Sources** | 你收集的源文档（文章、论文、图片、数据文件），不可变 | 你 |
| **Wiki** | LLM 生成的 Markdown 文件（摘要、实体页、概念页、综述） | LLM |
| **Schema** | 配置文件（如 `CLAUDE.md`），定义 wiki 结构和操作规范 | 你 + LLM 共同迭代 |

---

## 三种核心操作

### 1. 摄取（Ingest）

你把新来源放入 `raw/`，LLM：

1. 阅读来源
2. 与你讨论关键要点
3. 创建摘要页
4. 更新相关的 10-15 个实体页/概念页
5. 更新 index.md
6. 追加记录到 log.md

### 2. 查询（Query）

你基于 wiki 提问，LLM：

1. 先读 index.md 定位相关页面
2. 阅读相关页面后综合回答，附引用
3. **有价值的回答沉淀为新页面**（不会消失在聊天历史中）

### 3. 巡检（Lint）

定期让 LLM 健康检查：

- 页面之间的矛盾
- 被新来源取代的过时信息
- 没有入站的孤立页面
- 被提及但缺少独立页面的概念
- 缺失的交叉引用
- 可通过搜索填补的数据缺口

## 导航机制

### index.md —— 内容目录

- 按类别组织（实体、概念、来源等）
- 每个页面列出链接 + 一行摘要 + 可选元数据
- 每次摄取时更新
- 查询时 LLM 先读 index 定位页面，再深入阅读

### log.md —— 操作日志

- 追加式（append-only），按时间记录
- 格式示例：`## [2026-04-10] ingest | Article Title`
- 可用简单命令检索：`grep "^## \[" log.md | tail -5`

## 适用场景

| 场景 | 说明 |
|------|------|
| **个人成长** | 追踪目标、健康、心理，结构化地认识自己 |
| **深度研究** | 数周/数月读论文、文章，构建带演进论点的综合 wiki |
| **读书伴侣** | 每章 filing，构建人物、主题、情节的关系网络 |
| **团队知识库** | 维护内部 wiki，由 Slack 线程、会议记录、项目文档喂养 |
| **竞争分析** | 系统化追踪竞争对手动态 |
| **尽职调查 / 旅行规划 / 课程笔记** | 任何需要长期积累和组织知识的场景 |

## 为什么有效

人类放弃 wiki 的原因：

> **维护负担增长得比价值快。** 更新交叉引用、保持摘要最新、标记矛盾、维护一致性 —— 这些簿记工作让人崩溃。

LLM 不会偷懒，不会忘记更新交叉引用，可以一次修改 15 个文件。**维护成本趋近于零，所以 wiki 能持续保持健康。**

## 与 Memex 的渊源

这个理念是对 Vannevar Bush 1945 年 **Memex** 构想的现代实现：

- 一个个人化的、主动策划的知识库
- 文档之间的关联和文档本身一样有价值
- Bush 没解决的问题是 **"谁来维护"**
- **LLM 解决了这个问题**

## 核心要点总结

1. **不要 RAG，要 wiki** —— 从"每次重新检索"转向"持续积累编译"
2. **知识编译一次，持续更新** —— 而非每次查询时重新推导
3. **人管方向，LLM 管苦力** —— 你策展来源、提问题；LLM 做所有簿记
4. **好的回答要沉淀** —— 不让知识流失在聊天历史中
5. **Schema 是灵魂** —— CLAUDE.md 让 LLM 成为纪律化的 wiki 管理员
6. **保持抽象，按需实例化** —— 这套模式不限定具体工具，按需取舍
